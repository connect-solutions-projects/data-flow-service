Decisões Técnicas — DataFlow

Este documento apresenta as principais decisões técnicas tomadas durante o desenvolvimento do DataFlow, incluindo justificativas e trade-offs.

1. Monorepo

Decisão: Organizar todos os serviços e bibliotecas em um único repositório.

Justificativa: Facilita refatorações coordenadas, builds integrados, reuso de código e governança uniforme. Ideal para projetos com prazo curto e necessidade de sincronização entre componentes.

2. .NET 9

Decisão: Utilizar .NET 9 como plataforma principal.

Justificativa: Versão mais recente com melhorias de performance, suporte a Minimal APIs, e recursos modernos de C# 12.

3. Clean Architecture

Decisão: Organizar o código em camadas (Domain, Application, Infrastructure).

Justificativa: Separação de responsabilidades, testabilidade, independência de frameworks e facilita manutenção a longo prazo.

4. CQRS com MediatR

Decisão: Implementar padrão CQRS usando MediatR.

Justificativa: Separação clara entre comandos e queries, facilita testes, reduz acoplamento e melhora a organização do código.

5. RabbitMQ + MassTransit

Decisão: Usar RabbitMQ para mensageria com MassTransit como abstração.

Justificativa: Desacoplamento entre serviços, processamento assíncrono, escalabilidade e MassTransit simplifica o uso do RabbitMQ.

6. PostgreSQL + EF Core

Decisão: PostgreSQL como banco de dados principal com Entity Framework Core.

Justificativa: PostgreSQL é robusto, open-source, suporta transações ACID e EF Core facilita o desenvolvimento com migrations automáticas.

7. Redis

Decisão: Utilizar Redis para cache e coordenação distribuída.

Justificativa: Performance para rate limiting, deduplicação, cache de resultados e coordenação entre instâncias.

8. OpenTelemetry

Decisão: Implementar observabilidade com OpenTelemetry.

Justificativa: Padrão da indústria, suporte a múltiplos backends (Prometheus, Grafana), rastreamento distribuído e métricas customizadas.

9. Docker Compose

Decisão: Containerizar toda a stack com Docker Compose.

Justificativa: Facilita desenvolvimento local, reprodução de ambiente, deploy simplificado e isolamento de dependências.

10. Nginx como Proxy Reverso

Decisão: Usar Nginx para terminar TLS e rotear requisições.

Justificativa: Performance, suporte a SSL/TLS, facilita configuração de múltiplos serviços e padrão da indústria.
